## What CLAUDE.md promises vs. what the code does

1. **Region-markup subsystem (High-severity drift, detailed above).** CLAUDE.md's webapp section describes, in specific detail, a live "Sheet 1 mode bar," `REGION_PALETTE`, `reference_regions.json`, two REST endpoints, a datum/origin tool, and a Sheet-2 "Marked View" intake type. None of this exists in the current `webapp/` code. Likely explanation: an earlier iteration of the UI had per-view crop+markup preprocessing; a later redesign moved to "upload the full overview sheet, no human markup" (per Sheet 1's own on-page copy) and CLAUDE.md was never updated to match. **Fixed in this change** — CLAUDE.md is corrected to describe the current upload-the-full-sheet flow and the stale paragraph is removed (not silently left to rot further).
2. **Everything else in CLAUDE.md's pipeline architecture section (stages 1–13, the build-sequencer stage 6.5, the HoleWizard5 opt-in) checked out as accurate against the code** — this was verified directly against `main.py`'s actual call order and `pipeline/batch.py`'s `process_drawing_data`, not just re-read from the doc. No further drift found there.
3. **The guiding principle** ("a complete approximate model is always the correct outcome") is upheld by `resolver._completeness_gate` and `build_sequencer.py` as designed — but, as noted above, nothing previously *verified* that upheld principle against the original extraction after the fact. That is exactly what Stage 5 (Deliverable 3) now does.
