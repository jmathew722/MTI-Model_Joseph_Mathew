# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

MTI 2D‚Üí3D pipeline: converts 2D engineering drawings (PDF/PNG/JPG/DWG/DXF/eDrawings) into SolidWorks 2024 parts via Claude Vision extraction ‚Üí ambiguity resolution ‚Üí verification ‚Üí VBA macros ‚Üí `.sldprt`/`.stl` (COM, Windows-only). All code lives in `2D-3D-CAD-Test-Generation/`; sibling top-level folders (`Test2/`, `E2ETest/`, `DrawingPDFs/`, etc.) are test drawing sets.

## Commands

`python` is often NOT on PATH on this machine ‚Äî use the webapp venv interpreter:
`2D-3D-CAD-Test-Generation\webapp\.venv\Scripts\python.exe`.

All commands run from `2D-3D-CAD-Test-Generation/`:

```powershell
# Tests (conftest.py adds the project root to sys.path)
python -m pytest tests/ -q
python -m pytest tests/test_multiview.py -q          # single suite

# Golden macro snapshot (tests/test_golden_macros.py): after an INTENTIONAL
# change to macro output, regenerate and review the diff like code
$env:UPDATE_GOLDEN=1; python -m pytest tests/test_golden_macros.py -q

# Full pipeline on a batch of part folders (the standard test-batch command)
python main.py --views-folder ..\Test2 --output ..\Test2\output

# Single drawing; extract+verify only (no SolidWorks, cheapest live check)
python main.py --drawing path\to\drawing.pdf --validate-only --debug

# Rebuild from a saved extraction ‚Äî zero API cost
python main.py --from-json <Part>_extraction.json --output .\output

# Web UI (FastAPI on http://127.0.0.1:8092/) ‚Äî creates .venv + installs pinned deps on first run
cd webapp; .\run.ps1
```

Useful `main.py` flags: `--no-sldprt` (macros/reports only), `--no-export` (skip copy to `~/Downloads/SolidWorksModel_Parts`), `--no-extract-cache` (force paid re-extraction), `--skip-overview-check` / `--skip-requirements-check` (bypass the final READY gates), `--strict-gate` (block on failing verification). Exit codes: 0 = all parts READY, 8 = completed but not all READY, 2 = bad args.

Alternate entry points: `run_models.py <folders> --output ./output` (batch driver: extract ‚Üí auto-resolve ‚Üí build NON-STRICT so the run always completes, writing `<Part>_design_decisions.txt` per part); `build_sldprt.py <output_root>` (COM-builds every saved `<part>_extraction.json` under a root into `.sldprt`, one shared SolidWorks session, zero API cost).

Setup: `python setup.py`, then put `ANTHROPIC_API_KEY` in `.env` (gitignored). Optional env: `EXTRACTION_MODEL` (default `claude-sonnet-5`), `SOLIDWORKS_TEMPLATE_PATH` (must point at a real `.prtdot`), `MAX_IMAGE_LONG_EDGE`.

**Web UI dev gotcha:** after editing `webapp/app.py` or pipeline code, restart the uvicorn server on 8092 ‚Äî it otherwise serves stale endpoints.

## Architecture

### Pipeline (`2D-3D-CAD-Test-Generation/pipeline/`, orchestrated by `main.py`)

Stage order, one module per stage:

1. `utils/image_prep.py` ‚Äî normalize/downscale input images.
1.2. `utils/tiled_extraction.py` ‚Äî Workstream 2 "Tiled high-res extraction (zoom pass)" (2026-07-11): an ESCALATION (not the default) for large-format sheets whose thin line work dilutes to sub-pixel at the standard raster cap (the "image appears nearly blank" case). `should_tile()` fires on ANY of: the blank heuristic, ink density < 0.5%, extraction confidence < 0.6, > 25% of dims flagged unclear, or a C-size+ page at the ‚â§2576px cap. Then: `adaptive_render()` re-renders the VECTOR PDF at escalating DPI (300‚Üí600‚Üí900) until median line width ‚â• 2.5px (lossless zoom, never upscales a raster); a cheap global pass maps views/title-block/datum candidates; `make_tiles()` cuts ~1500px tiles with 22% OVERLAP; each content-bearing tile is extracted in SHEET coordinates (blank margins skipped for cost); `stitch()` merges by anchor+value (conflicting readings kept as candidate `possible_values` for Stage 2.5); `datum_anchor()` re-expresses positions from the datum (feeds Workstream 3). Tile cost logs separately as `extraction_tiled`; tiles cache by (page hash, DPI, grid). The VLM calls are injected so the machinery is unit-tested without paid calls; clean small drawings keep the single-shot path.
1.5. `overview_analysis.py` ‚Äî Stage 1.5 "Holistic Overview Analysis": the FULL uncropped sheet (the Tab-1 "FULL OVERVIEW VIEW" image / `00_full.jpg`) goes to Claude Sonnet 5 with a RELATIONAL prompt (views on the sheet, cross-view feature correspondences like through-vs-blind, overall 3D shape, cross-view conflicts with severity+recommendation, symmetry, global notes like "(6) HLS" with `resolved_count`) ‚Äî it does NOT re-extract dimensions. Output `overview_analysis.json`; fed to Stage 2.5 as **priority tier 2** (tier 0 = must-meet specs, tier 1 = per-view extraction owns dimension values, tier 2 = overview owns cross-view relationships); every resolution/flag records `resolved_by_tier`. Conflicts + a deterministic callout-count cross-check become flags (`source: overview_analysis`, e.g. the A050211E 5-visible-vs-"(6) HLS" case ‚Üí CRITICAL). Purely additive: no key / failure ‚Üí stage skipped, pipeline unchanged; `--from-json` reuses a sibling `overview_analysis.json`. Token cost is its own ledger line, `stage: stage_1_5_overview_analysis`. Shown in the UI as the collapsible "Overview Analysis" panel under the Full Overview View render.
2. `extractor.py` ‚Äî one Claude Vision call per part (all views labeled, forced tool call, Pydantic v2 schema in `schema.py`). **Specs-first:** operator must-meet specifications are injected into the extraction prompt so the model actively looks for those features from the start; the spec text is part of the cache key (changed specs force a fresh extraction). Raw extraction JSON is always saved; an on-disk cache (`<output>/.extraction_cache/`) makes identical re-runs free. Token/USD costs append to `token_usage_log.txt` via `usage_log.py`.
3. `vector_extract/` + `hole_resolution.py` ‚Äî exact hole positions from the original vector file (DXF/DWG entities via ezdxf, vector-PDF B√©zier circles via PyMuPDF, HoughCircles raster fallback). **Precedence rule: vector geometry owns position, the vision callout owns semantics (diameter/thread/depth); disagreement keeps both and flags CRITICAL.** Each hole carries `position_source` and `position_confidence`.
4. `must_meet.py` ‚Äî Stage 2.6 "Spec Reconciliation": the operator's MUST-MEET text (`must_meet_spec.txt`, the amber box in the UI; legacy `notes.txt` still works) is parsed into structured `MM-xxx` constraints (`must_meet_constraints.json`) via a dedicated Claude call with a deterministic regex fallback (no key needed). **Constraints are priority tier 0 ‚Äî they override vision-extracted values on any conflict**; every conflict goes to `lessons_learned.jsonl` (`resolution: spec_override`), never silently dropped. Missing geometry parameters are derived from extraction (bolt-circle fit: radius = mean ‚àö(x¬≤+y¬≤) about the hole centroid; radii disagreeing > 0.005 in = CRITICAL). Exception: a spec hole COUNT contradicting explicitly dimensioned drawing positions keeps the drawing's geometry, flags CRITICAL, and lets the MM check fail with measured-vs-required (`spec_vs_drawing_disagreement`).
5. `resolver.py` ‚Äî Stage 2.5, the core design decision: **the pipeline never blocks on ambiguity**. Every unclear dimension gets a numeric `resolved_value` chosen from extracted candidates (never fabricated) via a deterministic tree (spec-driven ‚Üí arithmetic chain ‚Üí geometric validity ‚Üí conservative geometry ‚Üí last-resort default), tagged HIGH/MEDIUM/LOW/CRITICAL. **Specs-first:** an operator must-meet spec value that clarifies an ambiguous reading takes precedence (Step 0, `assumption_basis="spec_driven"`). Stage 2.6's derived values + conflicts land in `resolved_extraction.json` under `must_meet`. **Commit-to-extraction mode (2026-07-11, `commit_mode` default ON; see `docs/commit-to-extraction-2026-07-11.md`):** no human in the loop ‚Äî the pipeline commits to the extraction and BUILDS every extracted feature; `EXCLUDED_INCOMPLETE` and `needs_markup_review` are no longer terminal for buildable geometry. **Positional dimensions are consumed before any escalation** (`_feature_positional_xy`) ‚Äî the Bug-1 root cause was that positional `applies_to` labels (`slot_offset`/`hole_position_x`/`position`) canonicalize to `""`, so a fully-extracted location (158-C `D002=1.56`) was dropped on the floor and the disposition recorded `[0,0]`/`needs_markup_review`. A step/notch cut missing length+width has its rectangle **derived from the outer-profile envelope minus its partial anchor dims** (`_derive_profile_delta`, `basis="profile_delta"`, M_121-B F002/F003); a hole missing a diameter inherits the most-common sibling diameter (`_sibling_diameter`, M_121-B F005 ‚Üí `.422`); a genuinely-undimensioned size/position commits a declared-basis conservative value/placement (`committed_conservative`, never `[0,0]`), applied, built, and CRITICAL-flagged with the value + basis + empty rungs. `commit_mode=False` restores the old exclude/review behavior for comparison. The only remaining hard failure is the existing one (no closed outer profile). Non-committable edge/pattern treatments (a fillet/chamfer with no size, a pattern with no count+spacing) stay excluded under both modes rather than fabricate a cosmetic value.
6. `validator.py` ‚Äî arithmetic/envelope verification, advisory by default (`--strict-gate` makes it blocking).
6.5. `build_sequencer.py` ‚Äî Stage 6.5 "Canonical build sequencer" (2026-07-10 redesign; see `docs/build-order-redesign-2026-07-10.md`): the ONE deterministic build-order pass. Called once at the top of `generate_macro_package`, it re-orders the completeness-gate survivors into a fixed **seven-stage** sequence (0 reference ¬∑ 1 base solid, largest closed profile ¬∑ 2 additive bosses ¬∑ 3 profile subtractions ¬∑ 4 holes: plain‚Üícbore/csk‚Üítapped ¬∑ 5 patterns ¬∑ 6 chamfers-then-fillets ¬∑ 7 non-geometric) with a **stable within-stage sort keyed to the feature id ‚Üí byte-identical `build_order` across runs**. **No type-based omission:** every feature ends in one of three states in `<Part>_build_dispositions.json` (also under `build_plan.json`'s `dispositions`) ‚Äî `BUILT`, `BUILT_WITH_DERIVED_VALUE` (resolver-inferred value), or `EXCLUDED_INCOMPLETE` (gated out with the missing parameter named; in commit-mode this is reached only by non-committable edge/pattern treatments). `_feature_xy` is slot-aware (reports a slot's true near-corner, never a `[0,0]` placeholder). Because the same `model` object flows onward, macros, `build_plan.json`, CadQuery pre-validation, and the COM build all inherit this order. **Bug-1 invariant (`macro_generator._assert_no_dropped_positions`):** generation REFUSES (`MacroGenerationError`) to emit a build whose disposition marks a position unresolved (`needs_markup_review`) while a positional dimension for that feature exists ‚Äî that dropped-position combination crashes loudly in tests instead of shipping a wrong part.
7. `macro_generator.py` + `macro_audit.py` ‚Äî numbered VBA macros (`00_setup` ‚Ä¶ `ZZZ_export_stl`, `RUN_ALL.vba`); prohibited features (loft/sweep/shell/etc.) become `NN_Fxxx_MANUAL_*.vba` steps; every macro is statically audited before writing (banned APIs fail generation). **Circular-pattern reliability layer:** a hole group routed to `circular_pattern` (spec says so, or the drawing dimensions polar-style; requires a concentric bore to derive the axis) emits three macros ‚Äî seed hole (`Fxxx_SeedHoleCut`) ‚Üí named reference axis (`PatternAxisN`, `InsertAxis2` from the bore cylindrical face) ‚Üí the pattern via the single `CreateCircularPatternSafe` VBA helper (version-pinned `FeatureCircularPattern5` signature from the installed sldworks.tlb, fallback to `...4`; axis Mark=1, seed Mark=4; Nothing-check + hard stop). `total_instances` INCLUDES the seed ‚Äî asserted once in the canonical build-plan schema (`circular_pattern` step; generation REFUSES if any canonical field is null). Every feature outcome is appended to `logs/macro_result.json` (JSON Lines) by the macros. **Workstream 3 "reference-geometry datum skeleton" (2026-07-11, `reference_geometry.py`):** BEFORE any feature, `01a_reference_geometry.vba` builds the drawing's datum structure as NAMED SolidWorks reference geometry ‚Äî `REF_DATUM_A` (base datum, always present) + `REF_DATUM_B/C` (from GD&T/dimension `datum_ref`) + `REF_SYM_X/Y` (symmetry) + `REF_AXIS_*` (concentric/circular) + `REF_PT_<fid>` (pattern origins), via `InsertRefPlane`/`InsertAxis2` (blacklist audit passes them). `build_plan.json` gains a `reference_geometry[]` block and each feature step a `positioned_from` handle. ADDITIVE ‚Äî the proven absolute-coordinate feature build (validated by the geometric loop) stays as the audit trail + fallback; the skeleton gives human landmarks and the deferred-retry loop stable named selection handles. codestack (`third_party/`, gitignored) mined for idioms; SW 2024 signatures in `docs/sw_api_reference/`. **Canonical slot / U-notch decomposition (2026-07-11, `slot_cut.py`):** a U-shaped cutout / open notch / slot is NEVER built as one arc-bearing sketch ‚Äî it is decomposed into exactly two ordered, adjacent build steps: **A. `slot_rect_cut`** ‚Äî a rectangular through-cut at the dimensioned position (`must_complete: true`; it carries the slot's position + size truth ‚Äî 4 lines / 4 dims is near-unfailable), immediately followed by **B. `slot_corner_fillet`** ‚Äî constant-radius fillets on the rectangle's interior corners (`defer_on_failure: true`, routed through the deferred-retry queue; a fillet failure never destroys the already-correct slot). The extraction schema gains `slot_cuts[]` (`SlotCut`: slot_kind open_notch/closed_slot/obround, open_edge, anchor_edge/anchor_offset + `anchor_semantics` edge_to_near_edge vs edge_to_centerline, width, depth, corner_radius, thru/thru_basis) ‚Äî additive, backward-compatible; the backing `extrude_cut` feature stays in `build_order` (the completeness gate skips any feature backed by a slot_cut, since its geometry lives on the slot record, not related_dimensions). `corner_array()` is the single source of truth both the rectangle sketch and the fillet edge-selection derive from (interior/near corners first). The resolver (Stage 2.5) validates fit (anchor+width ‚â§ extent), radius (2R ‚â§ width, R ‚â§ depth), anchor semantics, and through-all-from-single-view ‚Äî a geometry violation is CRITICAL with a ready-made clarification-gate question and is NEVER silently clamped; `normalize_legacy_slots()` converts a legacy extrude_cut+child-fillet pair into one canonical slot_cut. Macros: `_macro_slot_rect` (4 CreateLine + through-cut, hard-fail if the mandatory rectangle fails) then `_macro_slot_fillet` (vertex-proximity edge selection, asserts the expected corner count before `FeatureFillet3`, defers on mismatch). Reconciliation (`slot_checks`) flags a missing rectangle as CRITICAL and spot-checks the built near-edge X against the dimensioned anchor.
8. `cq_prevalidate.py` ‚Äî CadQuery pre-validation: builds the same geometry headlessly from `build_plan.json` (single source of truth; circular patterns via `.polarArray(radius, seedAngle, 360, count)` + `cutThruAll`), checks watertightness/volume/hole counts against the MM constraints, and writes `prevalidation.stl` + `prevalidation_report.json` + a per-run `prevalidate.py`. **A failed check aborts the SolidWorks build** and surfaces the exact constraint (`MM-001 FAILED: ‚Ä¶`). Graceful no-op when cadquery isn't installed.
9. `solidworks_builder.py` + `model_validator.py` ‚Äî Windows-only COM build of `.sldprt` + STL export + mass/bbox check. Same circular-pattern trio as the VBA path (`build_circular_pattern_holes`); features are renamed deterministically right after creation; per-feature outcomes are written to `macro_result.json` so a failure surfaces as the exact feature, never a generic exit code. **Optional `HoleWizard5` hole path** (`_try_hole_wizard`, OPT-IN via `MTI_ENABLE_HOLE_WIZARD=1`, default OFF): builds holes as real diameter-driven legacy wizard features at the resolved centers, with a verified fallback to the proven `_circular_cut_at` sketch-circle cut. The 27-arg signature is verified against the installed sldworks.tlb (dispid 222; no more "Type mismatch"), but SolidWorks 2024 returned `None` from the call even on a clean part, so the wizard is default-off until the version/locale-specific parameter/Value-slot mapping is nailed down on a live machine ‚Äî the default sketch-cut path is unchanged and regression-free. Everything upstream runs on any OS. **Workstream 1 "deferred feature retry" (2026-07-11, `deferred_retry.py`):** in non-strict mode a hard feature failure no longer skips forever ‚Äî it is QUARANTINED and the build CONTINUES; after the rest of the solid is complete, deferred features are retried (cap 3) with the completed-solid topology as context (their target faces/parents now exist), using an escalating taxonomy playbook (`classify_failure` ‚Üí selection / sketch-over-under-defined / zero-thickness-geometry / missing-parent / com-timeout / param-out-of-range; each attempt changes strategy, never repeats). A recovered feature becomes BUILT; a still-open one ends `deferred_open` with a ready-to-answer clarification question folded into the assist queue (never a silent skip). Ledger: `_deferred_log.json`.
10. `constraint_verify.py` ‚Äî post-build must-meet verification: the built STL is measured with trimesh (cross-section circle fitting; through-all = the hole appears near both faces) and every MM constraint is graded PASS/FAIL with measured vs required into `constraint_verification.json`. **A run with MM constraints is only READY when every constraint passes**; each failure is appended to `lessons_learned.jsonl` with the responsible VBA snippet.
10.5. `reconciliation.py` ‚Äî Stage 10.5 "Reconciliation Pass" (2026-07-10 self-correcting-loop upgrade; see `AUDIT_REPORT.md`/`CHANGE_PLAN.md`): the pipeline's own closing check of its output against the ORIGINAL raw `_extraction.json` ‚Äî never the resolved/downstream artifacts, which could themselves hide the bug. Builds a ground-truth checklist (every feature id + its expected instance count, from `hole_callouts[].qty`/`instance_positions`) and diffs it against the build-sequencer's disposition table + `build_plan.json`'s actual instance positions; a justified `skipped_prohibited` entry is accepted, anything else missing or short on instances is named exactly. On a gap, it re-runs **only** `resolve_extraction` (never the extractor ‚Äî no paid API call, the `--from-json`/cache cost discipline is untouched) with every requirements/overview-analysis signal freshly reloaded from disk, up to `max_passes` (default 3); since the resolver is a deterministic pure function, a pass that recovers nothing new stops the loop immediately rather than burning the cap. A recovered feature is spliced into the EXISTING `build_plan.json` (step replaced/appended) and a new `RECONCILE_pass<N>_*.vba` macro is added to `macros/` ‚Äî no existing file is renumbered or touched; the already-built `.sldprt` is not hot-patched (a full rebuild is needed to pick up the recovery ‚Äî noted in the report, not silently claimed). Writes `<Part>_reconciliation_report.json` (`checklist_total`, `confirmed_built`, `loop_passes_used`, `unresolved[]`, `splices_applied[]`, `final_status: READY | READY_WITH_OPEN_ITEMS`); any unresolved item folds into the engineering review (severity CRITICAL, source `reconciliation`) and gates the run's existing binary READY/NOT READY status (exit code 8) so nothing new had to be taught to the CLI/webapp's READY-banner parsing. Wired into both `batch.py::process_drawing_data` (the `--views-folder` path) and `main.py`'s single-drawing `--engine vba` path ‚Äî same guarantee regardless of entry point. Skipped (not an error) when `--no-resolve` was used, since there is no Stage 2.5 resolution to re-run against.
10.6. `feature_verify.py` ‚Äî Stage 10.6 "Per-feature geometric verification" (2026-07-10 accuracy layer): where `constraint_verify.py` grades the built STL against operator MM constraints only, this measures **every** planned feature against `build_plan.json` ‚Äî each hole's position + diameter + through/blind, each profile cut/notch's location (material-absence probe), each slot's obround, and the base envelope (+ a COM-vs-CadQuery volume cross-check). The STL is already in the lower-left-origin drawing frame (empirically verified). Every feature ends with one classification ‚Äî `OK` / `MISSING` / `MISPLACED` (measured position reported) / `WRONG_SIZE` (measured size reported) / `EXTRA` / `UNMEASURABLE` (always with a stated reason ‚Äî edge fillet/chamfer, cosmetic thread, non-watertight mesh) ‚Äî into `<Part>_feature_verification.json`. Reuses the proven cross-section machinery from `constraint_verify.py`; pure geometry, no ray/containment backend. A slot/cut's own boundary loop is recognised as its footprint, never a phantom EXTRA hole.
10.7. `reconciliation.py::geometric_correction_loop` ‚Äî Stage 10.7 "Geometric correction loop" (Phase B): wraps Stage 10.6 in a bounded build‚Üímeasure‚Üícorrect‚Üírebuild loop (cap 3, same discipline as the checklist reconciler). Correction policy by class: a **systematic** transform error (origin offset / axis swap / uniform scale ‚Äî detected only when ‚â•2 features share one consistent error, `classify_transform`) is corrected once and pre-compensated on every affected step; a one-off `MISPLACED` re-emits the single step with the **resolver-derived** position (the drawing is truth, never the measured value); `MISSING`/`EXTRA`/unresolvable `WRONG_SIZE` are flagged, never fabricated. Terminates on all-PASS (READY), the cap, no-applicable-correction (stops immediately, deterministic), or **oscillation** (a previously-PASS feature regressing ‚Üí stop, never thrash). Writes `<Part>_geometric_loop_report.json` and appends a `geometric_loop_iteration` entry to `lessons_learned.jsonl` each pass. The COM builder is injected (`build_fn`) so the loop is fully unit-tested without SolidWorks. Verified construction methods per feature class live in `pipeline/METHODS.md` (evidence-backed) + `methods_config.py` (machine dispatch, read by `macro_generator`; each build step records its `construction_method`); `construction_experiment.py` runs a scratch base+feature through candidate methods and Phase-A-verifies each to seed new library entries (holes ‚Üí `sketch_circle_cut`; slots ‚Üí `slot2d`/`create_sketch_slot`; both verified 2026-07-10).
10.8. `human_assist.py` ‚Äî Stage 10.8 "Human-assist escalation" (2026-07-10): the exit ramp when the automated ladder is genuinely exhausted. A feature/dimension becomes a question ONLY after all four automated stages fail (`escalation_eligible`): resolver plausibility ladder ‚Üí TYP/derivation ‚Üí the Phase B correction loop (3-pass cap) ‚Üí Phase D method experiments (chronic construction kinds only; N/A for a pure dimension ambiguity). Each eligible item becomes a NARROW question object (not an error dump): `question_text` (one sentence), pre-populated `candidates` with basis, a tight `region_crop`, and a `default_if_unanswered` that is **always populated** ‚Äî the best-available value that ships. Capped (default 3) and prioritized by leverage (a base/envelope dimension outranks one slot's radius; higher fan-out and CRITICAL tier rank higher). **Never blocks:** a pending question is a new *flagged* disposition overlay `NEEDS_HUMAN_INPUT` (additive on `build_dispositions.json`, geometric state unchanged) whose default still ships ‚Äî the part still produces its complete approximate model and its usual READY status; questions do NOT gate READY. Queue persists to `<Part>_assist_queue.json`; every Q (and eventual A) appends to `lessons_learned.jsonl`. Wired into both `batch.py` and `main.py` after reconciliation. **Answer feedback:** `apply_answers`/`rerun_with_answers` feed a human answer back as the resolver's highest-priority candidate (`assumption_basis="human_provided"`, `tier_human`, above spec) and re-splice the affected feature via the exact reconciliation splice-back ‚Äî **no paid re-extraction** (resolver only). The webapp exposes `GET/POST /api/parts/{session}/assist` (batched review, re-resolve on submit) and a Sheet-2 "Assist Needed" review queue (grouped by part, candidate radios + free-text override, default marked, count badge, empty-state-hidden ‚Äî easy to ignore, never a mandatory gate). `learned_patterns.py` generalizes a recurring ambiguity (same value-free `signature` answered the same relative way across ‚â•2 parts) into `LEARNED_PATTERNS.md` + `learned_patterns.json` ‚Äî a priority/confidence bias, never an auto-applied numeric value across drawings.
11. Final checks that gate READY (status only ‚Äî outputs are still produced): `overview_check.py` re-examines the part's overview drawing alone and diffs it against the build (missing visible feature = CRITICAL); `requirements_check.py` grades operator must-meet notes (`notes.txt` / `--requirements`) met/partial/unmet ‚Äî an unmet line gates READY. (Those same notes were already applied specs-first in stages 2 & 4; this is the final re-grade against the built part.)
12. `engineering_review.py` ‚Äî the single severity-ranked human report (`<Part>_engineering_review.txt`), regenerated after the COM build so skipped features are included. This is the canonical "what needs human attention" surface.
13. `learning_loop.py` ‚Äî the **iterative learning loop**: at the end of every run (`process_drawing_data`), reads the run's artifacts and writes one plain-text failure report to the repo-root `Learning Loop/<Part>__<timestamp>.txt` (gate reasons, MM constraint fails, Stage-1.5 conflicts, EVERY engineering-review flag at ALL severities CRITICAL/HIGH/MEDIUM/LOW, macro/build failures) ending with a paste-ready "FIXES FOR FABLE" brief (suspected code area per failure). A human hands these to Claude to plan generalized code fixes so the pipeline improves run over run; `Learning Loop/INDEX.md` logs one line per run. Exception-safe (never breaks a run).

Multi-view input (`--views-folder`): each part is a folder of per-view images; view role comes from filename keywords (`front`/`side`/`top`‚Ä¶) or a leading `01`‚Äì`05`; a file named `full`/`overview`/`isometric` or matching the folder name is overview context only (not built as a plane). Front + one other orthographic view required.

### Web UI (`2D-3D-CAD-Test-Generation/webapp/`)

`app.py` (FastAPI, port 8092) is a wrapper that **runs `main.py` as a subprocess** (`--views-folder <part> --output <part>/output`) and streams its console ‚Äî the CLI is the single source of truth for pipeline behavior. Frontend is a single `index.html` with four sheet-tabs (Sheet 1 crop ‚Üí Sheet 2 part setup/3D viewer ‚Üí Sheet 3 pipeline run controls + live Overview Analysis panel ‚Üí Sheet 4 run-outputs dock with the ten sub-tabs); `bridge.js` hooks the vendored DrawingCrop app (`photoapp/`, sources untouched) into the server. Sheet 2's "Select Model" dropdown and Sheet 4's "Select Run" dropdown are both backed by the single persistent `/api/run-history` endpoint (disk-scan of `webapp/parts/*/*/output` ‚Äî survives restarts and sessions); each run's console transcript is persisted to `ui_console.log` in the run output so Sheet 4's Console works for historical runs. **Pipeline Explainer (`webapp/explainer.py`, 2026-07-11) ‚Äî dual-provider read-only chat over ONE part's run artifacts:** answers questions ("why was D009 resolved this way?", "what's still deferred?", "where did the export go?") by grounding an LLM strictly in that run's artifacts. **Two providers, chosen per-message in the UI:** (1) **`local`** ‚Äî Ollama (qwen) on localhost, **zero cost**, private; every local request passes through `_urlopen` ‚Üí `assert_local`, which raises `ExternalHostError` before a socket opens for any non-localhost host (unit-tested at the socket layer), logs `cost_usd: 0.0`, sets `options.num_ctx: 16384` (Ollama's silent 2‚Äì4k truncation is the #1 way this would quietly break) and `think: false` (qwen thinking-models otherwise stream invisible reasoning first and appear to "output nothing"); an empty local answer surfaces an explicit error (the installed model may be too big for this machine's RAM) rather than a false success. ## Conventions

- Guiding principle (from the README): *a complete approximate model is always the correct outcome; an incomplete model is always the wrong outcome* ‚Äî resolve and flag, never block or silently drop.
- Numbers are chosen from extracted candidates, never invented; compliance grades are never fabricated (non-geometric notes ‚Üí `not_applicable`).
- Extraction JSON is backward-compatible: new fields are additive; old JSONs must keep loading (`--from-json`).
- Never commit `.env` or a real API key; `.env.template` stays keyless.
- **Reference-geometry naming contract** (Workstream 3): `REF_DATUM_<A|B|C>` (datum planes), `REF_SYM_<X|Y>` (symmetry mid-planes), `REF_AXIS_<purpose>` (centerlines/pattern axes), `REF_PT_<feature_id>` (pattern origins/anchors). These names are stable selection handles the deferred-retry loop relies on ‚Äî don't rename without updating both.
- **Feature end-states**: every extracted feature ends `built` / `built_with_flag` / `skipped_prohibited` / `deferred_open` (Workstream 1, rare ‚Äî ships with a clarification question, never a silent skip). Tiled extraction (Workstream 2) is an escalation, never the default; `third_party/` is gitignored (mining checkout, not a runtime dependency).
